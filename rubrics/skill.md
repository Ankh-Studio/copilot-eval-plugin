# Skill Evaluation Rubric

## Evidence-Based Scoring

Each criterion requires concrete evidence for scoring. Evaluators must provide specific examples
from the artifact that justify the score.

## Problem Definition (Weight: 0.25)

**Score 4**: Problem statement includes:

- Clear scope boundaries with specific deliverables
- Measurable success criteria with quantitative targets
- Identified constraints and assumptions
- Contextual background with stakeholder needs

**Evidence Required**: Explicit success criteria, scope boundaries, and constraint identification

**Score 3**: Problem statement includes:

- Defined scope with minor ambiguities
- General success criteria without specific metrics
- Some constraints identified
- Basic context provided

**Evidence Required**: Scope definition and success criteria present but may lack specificity

**Score 2**: Problem statement includes:

- Vague scope with unclear boundaries
- Success criteria not clearly defined
- Limited constraint identification
- Minimal context

**Evidence Required**: Basic problem statement but missing key elements

**Score 1**: Problem statement includes:

- Poorly defined scope
- No clear success criteria
- Missing constraints or context
- Unclear objectives

**Evidence Required**: Problem statement exists but lacks essential elements

**Score 0**: No clear problem statement or objectives

**Evidence Required**: No problem statement present

## Solution Completeness (Weight: 0.25)

**Score 4**: Solution demonstrates:

- Complete coverage of all problem requirements
- Edge case handling with error management
- Integration points and dependencies addressed
- Scalability and maintenance considerations

**Evidence Required**: All requirements addressed, edge cases handled, error management present

**Score 3**: Solution demonstrates:

- Coverage of most requirements with minor gaps
- Basic error handling for common scenarios
- Some integration considerations
- Limited scalability planning

**Evidence Required**: Majority of requirements covered, basic error handling

**Score 2**: Solution demonstrates:

- Partial coverage of requirements
- Minimal error handling
- Limited integration planning
- No scalability considerations

**Evidence Required**: Some requirements addressed but significant gaps exist

**Score 1**: Solution demonstrates:

- Minimal coverage of core requirements
- No error handling
- No integration or scalability planning

**Evidence Required**: Basic solution that barely addresses the core problem

**Score 0**: Incomplete or non-functional solution

**Evidence Required**: No functional solution provided

## Code Quality (Weight: 0.20)

**Score 4**: Code exhibits:

- Consistent formatting and naming conventions
- Comprehensive documentation with examples
- Unit tests with >80% coverage
- Performance optimizations and monitoring

**Evidence Required**: Style consistency, documentation, test coverage, performance metrics

**Score 3**: Code exhibits:

- Generally consistent formatting
- Good documentation with examples
- Unit tests with 50-80% coverage
- Basic performance considerations

**Evidence Required**: Mostly consistent style, documented, moderate test coverage

**Score 2**: Code exhibits:

- Some formatting inconsistencies
- Basic documentation
- Limited test coverage (<50%)
- No performance considerations

**Evidence Required**: Functional code with quality issues

**Score 1**: Code exhibits:

- Poor formatting and inconsistent naming
- Minimal or no documentation
- No tests
- Performance issues

**Evidence Required**: Code that hinders understanding or modification

**Score 0**: Code that is difficult to work with or understand

**Evidence Required**: Unreadable or unmaintainable code

## Tool Integration (Weight: 0.15)

**Score 4**: Integration demonstrates:

- Optimal tool selection for specific tasks
- Parallel processing and automation
- Error handling and recovery mechanisms
- Performance monitoring and optimization

**Evidence Required**: Appropriate tool usage, automation, error handling, performance optimization

**Score 3**: Integration demonstrates:

- Good tool selection with effective usage
- Some automation and parallelization
- Basic error handling
- Limited performance monitoring

**Evidence Required**: Effective tool usage with some automation

**Score 2**: Integration demonstrates:

- Basic tool usage with missed opportunities
- Minimal automation
- Limited error handling
- No performance considerations

**Evidence Required**: Basic tool usage without optimization

**Score 1**: Integration demonstrates:

- Minimal or inappropriate tool usage
- No automation or error handling

**Evidence Required**: Ineffective tool integration

**Score 0**: No meaningful tool integration

**Evidence Required**: No tool integration present

## Reusability (Weight: 0.15)

**Score 4**: Design demonstrates:

- Generic parameters and configuration options
- Modular architecture with clear interfaces
- Documentation for adaptation scenarios
- Examples for different use cases

**Evidence Required**: Configurable parameters, modular design, adaptation documentation

**Score 3**: Design demonstrates:

- Some configuration options
- Generally modular structure
- Basic adaptation documentation
- Limited examples

**Evidence Required**: Some reusability features with minor modifications needed

**Score 2**: Design demonstrates:

- Limited configuration options
- Some modularity but tight coupling
- Minimal adaptation documentation

**Evidence Required**: Limited reusability requiring significant changes

**Score 1**: Design demonstrates:

- Hardcoded values and tight coupling
- No modularity
- No adaptation documentation

**Evidence Required**: Too specific for practical reuse

**Score 0**: Single-use solution not designed for reuse

**Evidence Required**: No reusability considerations

## Validation Requirements

For each score above 2, evaluators must provide:

1. **Specific Evidence**: Exact quotes or references from the artifact
2. **Quantitative Metrics**: Numbers, percentages, or counts where applicable
3. **Cross-Reference**: How evidence maps to scoring criteria
4. **Confidence Level**: High/Medium/Low based on evidence strength

## Automated Validation

This rubric supports automated validation through:

- Pattern matching for evidence requirements
- Quantitative metric extraction
- Cross-reference validation
- Confidence scoring algorithms
